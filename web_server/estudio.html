Lógica de Control Exclusivo

El Arduino Nano actúa como el actuador principal. Es crítico comprender el concepto de exclusividad: la grúa solo puede ser movida por la web o por los joysticks físicos, nunca por ambos a la vez para evitar colisiones lógicas.

La conmutación se gestiona mediante la variable booleana controlWebActivo. Se alterna ante dos eventos: la pulsación del botón del Joystick 2 (pin A3) o el comando serie 'M'.

Sketch Arduino: Configuración SoftwareSerial
#include <SoftwareSerial.h>

// D12=RX, D9=TX para conectar con el ESP32
SoftwareSerial debugSerial(12, 9);

void setup() {
  Serial.begin(9600);      // Canal USB principal para telemetría
  debugSerial.begin(9600); // Canal auxiliar para depuración inalámbrica
  
  debugSerial.println("--- DIAGNÓSTICO INICIALIZADO ---");
}

void loop() {
  if (Serial.available() > 0) {
    char c = Serial.read(); // Leer comando desde USB
    if (c == 'M') {
      controlWebActivo = !controlWebActivo; // Alternar modo
      debugSerial.println("[MODO] Cambio por USB");
    }
  }
}

Explicación: Creamos un puerto serie emulado en pines D12/D9 usando interrupciones por software para enviar texto de diagnóstico hacia el ESP32, dejando el puerto de hardware ocupado exclusivamente con el canal USB de la laptop.

ESP32: Servidor de Logs en MicroPython
import uasyncio as asyncio
from machine import UART

uart = UART(2, baudrate=9600, rx=16, tx=17) # RX=16 conectado a D9 del Nano
log_buffer = []

async def read_debug_logs():
    buffer = ""
    while True:
        if uart.any():
            chunk = uart.read(uart.any())
            buffer += chunk.decode('utf-8', 'ignore')
            while "\n" in buffer:
                line, buffer = buffer.split("\n", 1)
                log_buffer.append(line.strip())
                if len(log_buffer) > 50:
                    log_buffer.pop(0) # Limitar a 50 líneas
        await asyncio.sleep_ms(20)

Explicación: Levantamos una corrutina asíncrona (`read_debug_logs`) en el ESP32. Esta tarea lee constantemente bytes de la UART2 conectados al SoftwareSerial del Arduino. Para proteger la RAM, se implementa una cola FIFO (First In, First Out) de máximo 50 líneas.

Frontend JavaScript: Web Serial API
let port;
async function connectSerial() {
    port = await navigator.serial.requestPort(); // Ventana nativa
    await port.open({ baudRate: 9600 });
    
    readLoop(); // Iniciar bucle asíncrono de lectura
}

async function readLoop() {
    const decoder = new TextDecoderStream();
    port.readable.pipeTo(decoder.writable);
    const reader = decoder.readable.getReader();
    
    let buffer = "";
    while (true) {
        const { value, done } = await reader.read();
        if (done) break;
        buffer += value;
        if (buffer.includes('\n')) {
            let lines = buffer.split('\n');
            buffer = lines.pop();
            for (let line of lines) {
                let data = JSON.parse(line.trim());
                processTelemetry(data); // Actualizar interfaz
            }
        }
    }
}

Explicación: La Web Serial API nos permite abrir el puerto USB directo en Chrome. El JavaScript consume un stream asíncrono y acumula cadenas en un buffer hasta detectar saltos de línea para decodificar las tramas JSON sin bloquear la UI.

El Divisor de Tensión (Acoplamiento de Niveles)

El Arduino Nano opera a un voltaje lógico de 5V. Sin embargo, los pines de entrada del ESP32 son sensibles y solo soportan un voltaje máximo de 3.3V. Si conectamos directamente el pin TX del Arduino (D9) al pin RX del ESP32 (GPIO 16), el chip del ESP32 podría dañarse.

Para solucionarlo, se implementa un divisor de tensión resistivo en la línea de transmisión del Arduino. El divisor reduce la amplitud de la señal eléctrica usando dos resistencias en serie conectadas a tierra:

Nano TX (5V) ----[ R1: 1kΩ ]----+---- GPIO 16 RX2 (3.3V)
                                |
                           [ R2: 2kΩ ]
                                |
                               GND

Cálculo Matemático:

V_out = V_in * (R2 / (R1 + R2)) = 5V * (2000 / (1000 + 2000)) = 5V * (2/3) ≈ 3.33V

Alimentación Separada (Fuerza vs Lógica)

Es muy importante mantener separadas las fuentes de alimentación:

  • Lógica (5V): Provista por el regulador del Arduino o el puerto USB de la laptop. Alimenta los sensores (joysticks) y la electrónica interna lógica de los controladores.
  • Potencia/Fuerza (9V): Provista por el transformador de fuente de poder externa de 9V a 2 Amperios hacia la bornera VM de los Puentes H. Los motores N20 demandan picos de corriente elevados al arrancar; si compartieran la línea lógica de 5V, provocarían caídas de tensión ("Brownouts") reiniciando al Arduino.

Arquitectura de Comunicación Dual

Para solucionar de forma robusta los problemas de almacenamiento y concurrencia del ESP32, se dividieron los roles del sistema físico de comunicación en dos canales independientes:

Canal USB de Control

Tecnología: Web Serial API

Propósito: Envío de comandos rápidos ('F','B','S') desde el navegador de la laptop y lectura del JSON de telemetría cada 200 ms.

Beneficio: Latencia prácticamente nula, alta fiabilidad física y no requiere correr servidores de backend.

Canal Wi-Fi de Diagnóstico

Tecnología: MicroPython Web Sockets/HTTP

Propósito: Exponer inalámbricamente los logs recopilados a través del puerto secundario SoftwareSerial del Nano.

Beneficio: Permite que un supervisor revise de forma remota en su celular qué órdenes recibe y ejecuta la grúa sin cables.

El Mecanismo de Seguridad: Parada por Timeout

En entornos industriales reales, una grúa torre que pierde comunicación con su operador no puede seguir moviéndose. En nuestro prototipo, implementamos este comportamiento mediante un latido de seguridad ("Watchdog timer"):

  1. Cuando la grúa está en Modo Web, el navegador de la laptop envía el carácter del comando activo de forma repetida cada 250 ms (ej: 'F' para avanzar carro).
  2. El Arduino Nano recibe el comando y actualiza la variable de temporización ultimoComandoWeb = millis().
  3. En cada ciclo del programa, el Arduino comprueba si el tiempo transcurrido desde el último comando supera los 500 ms: 
    if (millis() - ultimoComandoWeb > 500) { comandoWeb = 'S'; detenerMotores(); }
  4. Si el cable USB se desconecta, o el navegador se cierra, los motores se detienen automáticamente antes de transcurrir medio segundo.

💻 [Teoría] ¿Qué es un buffer circular y por qué se usa en el ESP32 para los logs?

Respuesta: Es una estructura de datos de tamaño fijo en RAM (en nuestro caso, una lista de máximo 50 elementos) que guarda los datos nuevos en orden secuencial. Cuando se llena, el elemento más antiguo se elimina automáticamente para dejar espacio al nuevo. Se usa en el ESP32 para evitar que la memoria RAM colapse por sobrecarga al almacenar logs continuos del Arduino Nano.

💻 [Teoría] En MicroPython, ¿qué ventaja ofrece usar "uasyncio" para el servidor de depuración?

Respuesta: Permite la multitarea cooperativa. El ESP32 puede leer de la UART (logs de Arduino) y al mismo tiempo escuchar peticiones de red HTTP en el puerto 80 sin bloquearse mutuamente. Si se hiciera de forma síncrona tradicional, la lectura de la UART detendría el servidor web y viceversa.

💻 [Teoría] ¿Por qué usamos "TextDecoderStream" y "getReader()" en el Javascript de la Web Serial API?

Respuesta: La Web Serial API entrega flujos de datos binarios (Uint8Array). `TextDecoderStream` convierte automáticamente esos bytes de entrada en texto decodificado UTF-8. El método `getReader()` permite crear un lector asíncrono para consumir ese texto línea por línea mediante bucles sin bloquear el hilo principal de la interfaz web.

💻 [Teoría] ¿Cómo se gestiona el control exclusivo de la grúa en el Arduino?

Respuesta: Mediante una bandera booleana `controlWebActivo`. Si es `true`, el Arduino ignora los valores analógicos leídos de los Joysticks y solo obedece a las órdenes serie provenientes del cable USB (Web). Si es `false`, se ignoran los comandos de la web y se leen los Joysticks físicos. La bandera se conmuta presionando el botón SW de Joystick 2 o enviando un comando Web especial.

💻 [Teoría] ¿Cómo funciona el filtrado de "zona muerta" en los Joysticks?

Respuesta: Los potenciómetros de los joysticks devuelven lecturas entre 0 y 1023. En reposo, deberían dar 512, pero el ruido eléctrico hace que oscilen (ej. 508, 517). Si aplicáramos movimiento directo, la grúa se movería sola. El código define una zona muerta entre 400 y 600; si el valor analógico cae dentro de este rango, la velocidad se establece en 0, deteniendo el motor.

🔌 [Teoría] ¿Cómo funciona el circuito integrado TB6612FNG y por qué es superior al antiguo L298N?

Respuesta: Es un Puente H doble basado en transistores MOSFET. Controla el sentido de giro de dos motores de CC mediante señales lógicas y regula la velocidad modulando la tensión con PWM. Es superior al L298N porque genera pérdidas de voltaje internas mínimas (baja resistencia RDS_on) y no requiere disipadores térmicos masivos, siendo altamente eficiente para baterías y fuentes pequeñas.

🔌 [Teoría] ¿Por qué los pines lógicos del ESP32 se dañan si se conectan a una salida de 5V de Arduino y cómo lo previene el divisor resistivo?

Respuesta: Las entradas del ESP32 operan a 3.3V lógicos y tienen compuertas de óxido de silicio muy delgadas que pueden perforarse por sobretensión si reciben 5V directos. El divisor de tensión resistivo (resistencia de 1kΩ en serie con la señal y 2kΩ a tierra) reduce el voltaje en una relación 2/3, entregando ~3.33V seguros a la entrada del ESP32.

🔌 [Teoría] ¿Cuál es el propósito del pin Standby (STBY) del TB6612FNG y cómo se conecta?

Respuesta: El pin STBY apaga por completo las salidas lógicas de fuerza del puente H para ahorrar energía y evitar corrientes residuales accidentales en los motores. Debe recibir un estado lógicamente alto (5V) para que el integrado funcione. Se conecta comúnmente a 5V lógicos o a un pin digital del Arduino configurado como OUTPUT en HIGH.

🔌 [Teoría] ¿Por qué los potenciómetros de los joysticks requieren una conexión a la línea VCC lógicos y GND?

Respuesta: Los potenciómetros actúan como divisores de tensión variables. Al alimentarse con VCC en un extremo y GND en el otro, el pin central (limpiador) varía su voltaje entre 0V y VCC en función de la posición física del joystick. Este voltaje variable es lo que lee el conversor analógico-digital (ADC) del Arduino Nano.

🔌 [Teoría] ¿Qué es la tierra común (GND común) y qué sucedería si se omite entre el ESP32 y el Arduino?

Respuesta: Es la unificación del punto de referencia de voltaje de todas las fuentes de alimentación del circuito. Si se omite la conexión de tierra común entre el Arduino y el ESP32, las señales de comunicación serial SoftwareSerial (TX/RX) carecerán de un voltaje de referencia estable, lo que provocará la pérdida absoluta de datos, ruidos y corrupción total de los logs.

📐 [Teoría] ¿Por qué se prefiere la Web Serial API en lugar de levantar un WebSocket de control en el ESP32?

Respuesta: Porque el ESP32 tiene recursos limitados de CPU y RAM. Correr un servidor WebSocket de alta frecuencia para procesar comandos de control de motores y enviar telemetría satura su pila de red. El cable USB mediante Web Serial API ofrece una conexión por cable redundante, latencia de 1 ms, sin caídas de conexión por interferencia de RF, liberando al ESP32 para que actúe únicamente como monitor pasivo.

📐 [Teoría] ¿Cómo se estructuró la distribución de archivos y por qué se limpió el directorio raíz?

Respuesta: Se clasificó el proyecto en `/arduino/` (firmware principal), `/esp32/` (depuración inalámbrica) y `/web_server/` (archivos frontend). Se limpió el directorio raíz para evitar que el compilador de Arduino o el intérprete de ESP32 cargaran archivos obsoletos u homónimos por error, asegurando la mantenibilidad e integridad del repositorio.

📐 [Teoría] ¿Por qué la interfaz web de control (index.html) se ejecuta de manera local en el navegador en lugar de servirse desde el ESP32?

Respuesta: Porque la interfaz web incluye hojas de estilos, tipografías y gráficos vectoriales SVG pesados para representar la grúa torre en tiempo real. Servir estos activos estáticos desde la memoria flash del ESP32 consume demasiado tiempo y memoria, lo que provocaba reinicios del microcontrolador. Al cargarlo en la laptop localmente, el navegador de la laptop procesa los assets y se comunica con el hardware por USB.

📐 [Teoría] ¿Cuál es el beneficio de separar las fuentes de alimentación lógica (5V) y de potencia (9V)?

Respuesta: Los motores eléctricos de CC inducen picos de corriente elevados y ruido electromagnético en las líneas de alimentación al arrancar o cambiar de dirección. Al separarlas, la fuente lógica de 5V del Arduino permanece limpia y estable, evitando caídas de voltaje críticas ("brownouts") que reiniciarían el microcontrolador Nano.

📐 [Teoría] ¿Cómo influye el tamaño de la pluma y la estabilidad del contrapeso en el diseño físico de la grúa?

Respuesta: En el diseño de una grúa torre, la pluma (donde se mueve el carro) y la contrapluma (donde va el contrapeso) deben balancearse. El momento mecánico (Fuerza x Distancia) generado por el contrapeso debe equilibrar el peso de la pluma y la carga máxima del carro. Si no se balancea en el diseño estructural, la grúa sufrirá volcamiento o pandeo de la torre.

🧪 [Teoría] ¿Qué es el "Watchdog por Software" (parada por timeout) implementado en el Arduino Nano?

Respuesta: Es una rutina de control de tiempo en el código. El navegador envía de forma periódica un comando activo cada 250 ms. El Arduino guarda la marca de tiempo del último comando recibido usando `millis()`. Si el tiempo transcurrido supera los 500 ms (el doble del intervalo esperado), el Arduino detiene de inmediato los motores asumiendo que la laptop se desconectó o la interfaz web falló.
